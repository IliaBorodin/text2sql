# Анализ приложения Text2SQL

## 1. Назначение приложения

Приложение реализует сервис `text-to-SQL` для предметной области ЖКХ. Пользователь задаёт вопрос на естественном языке на русском языке, после чего система:

1. находит релевантные таблицы схемы;
2. собирает контекст по структуре БД;
3. отправляет вопрос и контекст в LLM;
4. извлекает SQL из ответа модели;
5. валидирует запрос;
6. выполняет его в PostgreSQL в режиме `read only`;
7. возвращает SQL, результат выполнения, список использованных таблиц и тайминги.

По архитектуре это API-сервис на `FastAPI`, работающий как оркестратор вокруг трёх внешних зависимостей:

- `Ollama` для генерации SQL и эмбеддингов;
- `ChromaDB` для векторного поиска по схеме;
- `PostgreSQL` для выполнения итогового запроса.

## 2. Архитектурная схема

Код разделён на понятные слои:

- `app/api` — HTTP-роуты (`/query`, `/health`);
- `app/services` — бизнес-логика пайплайна;
- `app/infrastructure` — работа с Ollama, PostgreSQL, ChromaDB и `.xdic`;
- `app/core` — настройки, модели, интерфейсы и исключения.

Это хорошее решение: слой API почти не содержит логики, а основные зависимости инъецируются в `Pipeline` при старте приложения. За счёт этого код читаемый и пригоден для дальнейшего тестирования и замены адаптеров.

## 3. Как работает основной сценарий

### 3.1. Инициализация

При старте приложения создаются:

- `OllamaClient`;
- `PostgresClient`;
- `ChromaClient`;
- `XdicParser`;
- сервисы `SchemaRetrievalService`, `PromptBuilder`, `SQLGenerator`, `SQLValidator`, `SQLExecutor`;
- единый `Pipeline`.

Инициализация собрана в `lifespan` приложения, что делает жизненный цикл аккуратным: подключения открываются при запуске и закрываются при остановке.

### 3.2. Обработка запроса

Маршрут `/api/query` принимает `question`, передаёт его в `Pipeline.execute()` и возвращает сериализованный результат.

Внутри пайплайна происходит 7 фаз:

1. поиск релевантных таблиц;
2. сборка промпта;
3. генерация SQL;
4. валидация;
5. выполнение;
6. одна попытка самокоррекции при ошибке;
7. сборка ответа.

Это сильная сторона приложения: пайплайн прозрачен, логируется по шагам и возвращает измеримые тайминги, что удобно для эксплуатации и отладки.

## 4. Сильные стороны реализации

### 4.1. Удачная декомпозиция

Сервисы отвечают каждый за свой участок:

- `SchemaRetrievalService` ищет таблицы и расширяет контекст связанными сущностями;
- `PromptBuilder` собирает контекст для модели;
- `SQLGenerator` общается с LLM и извлекает SQL;
- `SQLValidator` страхует от опасных или некорректных запросов;
- `SQLExecutor` выполняет SQL;
- `Pipeline` управляет всем сценарием.

Это один из самых сильных аспектов проекта: код не монолитный, а роли компонентов хорошо разделены.

### 4.2. Практичная защита выполнения SQL

Защита реализована в нескольких местах:

- валидация через `sqlglot`;
- запрет `INSERT/UPDATE/DELETE/DROP/ALTER/...`;
- разрешение только `SELECT`-подобных выражений;
- автоматическое добавление `LIMIT`, если его нет;
- выполнение транзакции в `READ ONLY`;
- `statement_timeout` на уровне PostgreSQL;
- ограничение числа возвращаемых строк через `fetchmany`.

Для production-подобной системы это очень полезный слой безопасности. Даже если модель ошибётся, шанс нанести вред БД заметно снижен.

### 4.3. Хорошая работа с контекстом схемы

Схема берётся не только из векторного поиска. После первичного поиска сервис подтягивает связанные таблицы по внешним ключам и формирует список связей в человекочитаемом виде. Это повышает вероятность, что LLM построит корректный `JOIN`, а не просто угадает таблицу по названию.

Особенно полезно, что приложение использует `.xdic` как семантический слой предметной области, а не только «сырую» структуру PostgreSQL.

### 4.4. Наличие контура самокоррекции

Если SQL не прошёл валидацию или не выполнился в БД, пайплайн делает ещё одну попытку, передавая модели предыдущий SQL и текст ошибки. Это разумный и прикладной компромисс: система не зацикливается, но умеет исправлять типовые проблемы.

### 4.5. Есть вспомогательные скрипты эксплуатации

В проекте уже есть:

- `scripts/index_schema.py` для индексации схемы в ChromaDB;
- `scripts/test_queries.py` для прогона контрольных вопросов.

Это хороший знак: проект ориентирован не только на разработку, но и на поддержку реального процесса настройки качества.

## 5. Ограничения и риски

### 5.1. Слабое место: схема PostgreSQL фактически не фиксируется на уровне сессии

В `PostgresClient` сохраняется `db_schema`, но при выполнении SQL не видно `SET search_path` или другого механизма, который бы гарантировал работу именно в нужной схеме. Сейчас поле логируется, но само по себе не влияет на выполнение запроса.

Последствия:

- LLM должна сама указывать схему или надеяться на дефолтный `search_path`;
- одинаковые имена таблиц в разных схемах могут давать неожиданный результат;
- поведение может отличаться между окружениями.

Это один из главных архитектурных рисков.

### 5.2. DDL из `XdicParser` не полностью соответствует реальной БД

`get_create_table_sql()` генерирует полезный для LLM DDL, но он скорее «контекстный», чем точный:

- `row_id` всегда добавляется как `serial primary key`;
- внешние ключи описываются комментариями, а не полноценными `FOREIGN KEY`;
- реальные ограничения, индексы и точные типы могут быть отражены неполно;
- парсер при старте приложения не обогащается данными из живой БД.

Для генерации SQL этого часто достаточно, но точность контекста ограничена.

### 5.3. Векторный поиск полностью зависит от заранее построенного индекса

Если `ChromaDB` не синхронизирована с текущей `.xdic` или БД, качество поиска резко падает. В текущем виде нет автоматической проверки актуальности индекса и нет механизма его автообновления при изменении схемы.

### 5.4. В приложении почти не видны автоматические тесты как части CI

Есть полезный `scripts/test_queries.py`, но это скорее ручной или полуавтоматический прогон. По структуре проекта не видно unit/integration test-пакета, который регулярно проверял бы:

- корректность валидации SQL;
- извлечение SQL из ответов модели;
- работу пайплайна на моках;
- устойчивость к ошибкам инфраструктуры.

Для системы, завязанной на LLM и внешние сервисы, это важный пробел.

### 5.5. Контекст обрезается по символам, а не по смыслу

`PromptBuilder` ограничивает контекст простым порогом `_max_context_chars = 15000`. Это рабочий вариант, но у него есть недостатки:

- могут отрезаться важные таблицы в конце списка;
- порядок влияет на качество сильнее, чем хотелось бы;
- нет отдельного бюджета на связи, примеры значений и критичные поля.

На малых схемах это нормально, но на большой предметной модели будет ощутимым ограничением.

### 5.6. Самокоррекция полезна, но пока очень базовая

Сейчас используется одна повторная попытка с текстом ошибки. Не видно:

- раздельной обработки синтаксической и семантической ошибки;
- хранения истории неудачных попыток;
- анализа, какая именно таблица или колонка отсутствует;
- fallback-стратегий при низком качестве retrieval.

То есть механизм уже есть, но пока это минимальная версия.

## 6. Оценка качества ключевых модулей

### `Pipeline`

Центральный модуль реализован качественно: этапы читаемы, ошибки локализованы, ответ всегда имеет единый формат. Это хороший оркестратор.

### `SchemaRetrievalService`

Модуль практичный и полезный: сначала берёт семантически релевантные таблицы, затем расширяет набор через связи. Это повышает шанс на корректные `JOIN`.

Ограничение в том, что вес связанных таблиц принудительно становится `0.0`, а качество отбора зависит только от количества связей, а не от реальной полезности таблицы для вопроса.

### `SQLValidator`

Один из самых зрелых компонентов проекта. Он не только фильтрует опасные команды, но и умеет извлекать SQL из Markdown-ответа модели и даже пытается чинить часть проблем с кириллическими идентификаторами.

### `PostgresClient`

Решение аккуратное и безопасное по режиму исполнения, но хотелось бы сильнее контролировать контекст БД: схему, `search_path`, возможные ограничения по ролям и более явную защиту от тяжёлых запросов.

### `XdicParser`

Это важный доменный актив проекта. Он делает систему умнее за счёт описаний таблиц, связей и бизнес-контекста. При этом сам файл большой, смешивает несколько ролей и выглядит менее «приглаженным», чем остальная часть проекта. Это кандидат на последующий рефакторинг.

## 7. Что можно улучшить в первую очередь

### Приоритет 1

1. Явно задавать `search_path` или включать схему в генерируемый SQL-контекст.
2. Добавить автоматические тесты для `SQLValidator`, `SQLGenerator` и `Pipeline`.
3. Проверять актуальность индекса ChromaDB относительно `.xdic`.

### Приоритет 2

1. Улучшить стратегию отбора контекста: не просто по лимиту символов, а по важности таблиц и связей.
2. Обогащать `XdicParser` реальными метаданными БД хотя бы в offline-режиме при индексации.
3. Улучшить self-correction: разные подсказки для ошибок синтаксиса, отсутствующих колонок и неверных `JOIN`.

### Приоритет 3

1. Разделить `XdicParser` на несколько модулей: парсинг XML, построение контекста, генерация DDL.
2. Добавить наблюдаемость: метрики успешности генерации, ошибок валидации, частоты ретраев, типовых падений SQL.
3. Сохранять `explanation` от LLM в ответ API или в логи для диагностики.

## 8. Итоговая оценка

Проект производит хорошее впечатление как прикладная Text2SQL-система для узкой доменной области. Его главные плюсы:

- понятная слоистая архитектура;
- хороший пайплайн оркестрации;
- сильный слой защиты вокруг SQL;
- полезное сочетание векторного поиска и доменного `.xdic`;
- наличие практического контура самокоррекции.

Главные ограничения сейчас лежат не в общей идее, а в точности контекста и эксплуатационной зрелости:

- схема БД задаётся не до конца явно;
- контекст для LLM местами приближённый;
- тестовое покрытие и автоматическая верификация пока выглядят ограниченными;
- качество retrieval зависит от внешнего индекса без встроенного контроля актуальности.

Итог: приложение уже выглядит как рабочий MVP или ранний production-кандидат для внутреннего использования, особенно если схема БД стабильна и домен хорошо известен. Чтобы сделать систему устойчивой в долгой эксплуатации, в первую очередь стоит усилить контроль над схемой PostgreSQL, покрытие тестами и качество подготовки контекста для LLM.
